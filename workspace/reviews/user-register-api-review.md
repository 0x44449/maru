# user-register-api 설계 리뷰

## 결론

승인 가능. 이전에 지적한 `maru_id` 의미 충돌, 베이스라인 리셋 표현, 아바타 업로드 운영 메모, 파일 업로드 소유권 기준까지 반영됐다.

## 확인된 개선 사항

- `maru_id`를 "프로비저닝 시 자동생성, 이후 1회 변경 가능"으로 일관되게 고정했다.
- provision 응답 예시도 자동생성된 `maruId`를 반환하도록 맞췄다.
- 기존의 `DROP TABLE ... CASCADE` 기반 V2 마이그레이션 대신, 개발 전용 베이스라인 리셋 절차로 의미를 명확히 바꿨다.
- `(auth_provider, auth_provider_id)` 복합 조회, `X-Profile-Id` 실패 계약, `check-id` 상태 응답도 유지되고 있다.
- 아바타 업로드에서 매직 바이트 검사, 리사이즈본 생성, 상대 경로 저장, 원본 파일명 미노출, 지연 삭제 정책까지 문서화한 점이 좋다.
- 파일 업로드 API에 `X-Profile-Id`를 필수로 두고 `uploader_id = profile_id` 기준을 고정한 점도 일관적이다.

## 메모

- `V1__init_schema.sql` 교체 + `flyway_schema_history` 리셋은 설계상 이해 가능하지만, 팀 전체가 같은 시점에 개발 DB를 함께 초기화한다는 전제가 필요하다.
- 따라서 이 절차는 구현 전에 "개발 환경 전용"이라는 점을 실제 작업 순서와 README/운영 메모에도 같이 남기는 편이 좋다.

## 총평

문서 구조와 결정 사항은 충분하다. 자동 프로비저닝, 프로필 수정, 파일 업로드, MinIO 인프라, 베이스라인 리셋까지 큰 방향이 일관되게 정리돼 있다. 구현 진행해도 된다.
