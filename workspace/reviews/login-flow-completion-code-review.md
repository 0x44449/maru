# login-flow-completion 코드 리뷰

## 리뷰 기준

이번 리뷰는 현재 구현 범위를 기준으로 수행했다. `user-register` 완료 처리와 실제 사용자 등록 API 연동은 이번 범위에 포함하지 않는다. 이번 차수는 기능 적합성보다 다음 관점에 초점을 뒀다:

- 문자열/암묵적 규칙에 기대는 취약한 판별 로직 여부
- 성급한 공통화나 불필요한 추상화 여부
- 책임이 맞는 계층에 배치되었는지
- 컨벤션과 정리 상태

## 주요 지적 사항

### 1. 카카오 로그인 취소 판별은 여전히 문자열 패턴에 기대고 있음

- 관련 파일: `apps/mobile/src/libs/auth/index.ts`
- 관련 위치: `isCancelError()`
- 이번 수정으로 취소 판별 책임이 UI에서 auth 계층으로 내려간 점은 좋다. 다만 카카오 쪽은 아직 `e.message.includes("statusCode=-1")`에 의존한다.
- 이전보다 계층은 맞아졌지만, 판단 근거 자체는 여전히 SDK 메시지 포맷에 묶여 있다. 메시지 포맷이 바뀌면 곧바로 깨질 수 있다.
- 가능하면 Kakao SDK가 제공하는 코드/타입 기반 판별로 바꾸는 편이 좋다. 그게 불가능하다면, 최소한 이 문자열 의존이 왜 필요한지 주석으로 명확히 남겨둘 필요가 있다.

## 확인된 사항

- `login.tsx`에서 `handleLogin(provider)`를 제거하고 `handleKakaoLogin`, `handleGoogleLogin`으로 분리한 점은 더 직관적이다.
- 로그인 취소/실패 판별 책임이 UI에서 auth 계층으로 이동해 계층 구조가 좋아졌다.
- Root layout의 인증 상태 기반 라우팅은 단순하고 이해 가능하게 유지됐다.
- Zustand 인증 스토어 구조는 과도하게 복잡하지 않고 책임이 비교적 명확하다.
- `/me` 조회 실패 시 `error` 상태와 재시도 UI를 분리한 점은 유지보수 측면에서 적절하다.

## 결론

승인 가능. 이전 리뷰 포인트는 대부분 반영됐고, 남은 건 카카오 취소 판별의 근거가 아직 문자열 패턴에 기대고 있다는 점 정도다.
